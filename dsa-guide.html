DFS — Depth-First Search

One sentence: Go as deep as possible along one path, then backtrack and try the next branch — like walking a maze and always taking the first open turn until you hit a wall.

Real-life analogy

You're exploring a building floor plan with a pen. Enter a room, mark it visited, open the first door, keep going until every reachable room from that entrance is marked. That's one "connected component." Start again from any unmarked room for the next component.

Interactive trace — flood fill on a grid

In Number of Islands, each group of adjacent '1' cells is one island. DFS sinks each cell to '0' so you never visit twice.

Step 1 / ?

Minimal C skeleton

/* Grid DFS — 4 directions */
void dfs(char g[][COLS], int rows, int cols, int r, int c) {
    /* 1. BASE CASE — stop if out of bounds or not land */
    if (r < 0 || r >= rows || c < 0 || c >= cols) return;
    if (g[r][c] != '1') return;

    /* 2. MARK VISITED — sink so we don't revisit */
    g[r][c] = '0';

    /* 3. RECURSE — try all 4 neighbors */
    dfs(g, rows, cols, r + 1, c);
    dfs(g, rows, cols, r - 1, c);
    dfs(g, rows, cols, r, c + 1);
    dfs(g, rows, cols, r, c - 1);
}

Line-by-line trace (first call dfs(g, 4, 5, 0, 0))

  1. Enter (0,0): in bounds, g[0][0]=='1' → set to '0'
  2. Recurse down: dfs(1,0) → same island, mark and continue
  3. Recurse right: from (0,0) also try (0,1), (0,2) … entire blob gets sunk
  4. Return: when all four recursive calls finish, this cell's DFS is done
  5. Outer loop (in main's helper): scan grid; next '1' at (3,3) starts a new island count
Use DFS when: counting connected regions, exploring all paths, tree recursion (left/right), or mutating the grid in place.
Tree example: Tree Path Sum (q120) — recurse left and right with a smaller target.

BFS — Breadth-First Search

One sentence: Explore in expanding rings — all cells 1 step away first, then 2 steps, then 3. The first time you reach the goal, you have the shortest path (on an unweighted grid).

Real-life analogy

Ripples in a pond. Drop a stone at the start — the wave hits nearby cells first. BFS uses a queue (FIFO: first in, first out) so cells are processed in order of distance.

Interactive trace — shortest path on a grid

Open cells expand layer by layer. A wall forces the path to go around. When the goal cell is dequeued, its distance is the answer.

Step 1 / ?

Minimal C skeleton

/* Queue arrays live outside or as static — but head/tail reset EACH call */
int q_r[MAX_CELLS], q_c[MAX_CELLS], q_d[MAX_CELLS];

int shortestPath(char g[ROWS][COLS], int goal_r, int goal_c) {
    /* FIX: reset head/tail inside helper — not as globals */
    int head = 0;
    int tail = 0;
    int vis[ROWS][COLS] = {0};

    q_r[tail] = 0; q_c[tail] = 0; q_d[tail] = 1;
    tail++;
    vis[0][0] = 1;

    while (head < tail) {
        int r = q_r[head];
        int c = q_c[head];
        int d = q_d[head];
        head++;

        if (r == goal_r && c == goal_c) return d;

        /* for each open neighbor: if (!vis[nr][nc]) { vis[nr][nc]=1; enqueue(nr,nc,d+1); } */
    }
    return -1;   /* goal unreachable */
}

Line-by-line trace

  1. Enqueue (0,0) dist=1 — queue is not empty, so BFS can run
  2. Dequeue front — process the oldest cell first (that's what makes it "breadth" first)
  3. Push neighbors with dist+1 only if unvisited — mark in vis[][]
  4. Goal check when dequeuing — not when enqueueing — so distance is correct
Use BFS when: shortest steps on a grid, level-order traversal, or "minimum number of moves."

DFS vs BFS — quick comparison

DFS

  • Stack (implicit via recursion) or explicit stack
  • Goes deep first
  • Great for: flood fill, all paths, trees
  • Does not guarantee shortest path

BFS

  • Queue (head / tail)
  • Expands layer by layer
  • Great for: shortest path on unweighted grid
  • Uses more memory at wide layers
Tree DFS

Trees — same DFS idea, different shape

One sentence: A binary tree is nodes with left and right pointers. DFS on a tree = recurse into children instead of grid neighbors.

Grid DFS visits 4 neighbors. Tree DFS usually visits 2 children. The base case is root == NULL. Leaf = both children are NULL.

Interactive trace — path sum to 22

Subtract each node value from the target as you go down. At a leaf, check if remaining is 0.

Step 1 / ?

Minimal C skeleton

struct Node { int val; struct Node *left, *right; };

int hasPathSum(struct Node *root, int target) {
    if (!root) return 0;
    if (!root->left && !root->right)
        return root->val == target;   /* leaf */
    return hasPathSum(root->left, target - root->val)
        || hasPathSum(root->right, target - root->val);
}

BST insert/search/traverse live in the Binary Search Tree category (q63–q70). Tree path sum is the bridge from grid DFS to pointer recursion.

Table / recurrence

Dynamic programming (DP)

One sentence: Save answers to smaller sub-problems in an array so you never recalculate them from scratch.

Real-life analogy

You write down "1+1+1+1+1 = 5" on paper. When someone adds another "+1", you don't count from the beginning — you use the 5 you already wrote and add 1 to get 6. DP is that scratch paper as an array.

How to think

DP is not a special library or keyword in C. It's a recipe:

  1. Define dp[i] in English: "number of ways to reach stair i" or "best loot using houses 0..i"
  2. Find the recurrence: how does dp[i] relate to smaller indices?
  3. Base cases: dp[0], dp[1] you know by hand
  4. Fill the table left to right (or nested loops for 2-D)

Interactive trace — Climbing Stairs (n=5)

Same problem, three implementations. Switch mode, then step through each.

Top-down + memo

Step 1 / ?

Minimal C skeleton (bottom-up table)

int climbStairs(int n) {
    if (n <= 2) return n;

    int dp[n + 1];
    dp[1] = 1;
    dp[2] = 2;

    for (int i = 3; i <= n; i++) {
        dp[i] = dp[i - 1] + dp[i - 2];   /* past answers, no recompute */
    }
    return dp[n];
}

Line-by-line trace

  1. Small n: return 1 or 2 directly for 0–2 steps (base cases).
  2. Allocate table: dp[n+1] holds every sub-answer up to stair n.
  3. Fill upward: loop from 3 to n; each dp[i] reads only dp[i-1] and dp[i-2] — no deep recursion tree.

Space-saving variant (2 variables)

int climbStairs(int n) {
    if (n <= 1) return 1;
    int a = 1, b = 1;   /* dp[0], dp[1] */
    for (int i = 2; i <= n; i++) {
        int c = a + b;
        a = b;
        b = c;
    }
    return b;
}

Memoization vs tabulation (both are DP):
Top-down — recursive function + cache array: "if dp[i] already computed, return it." Feels like DFS with memory.
Bottom-up — fill dp[0..n] in a loop (what q111–q113 use). Same math, often easier in C interviews.
Start with tabulation; switch to memo if the recurrence is naturally recursive (e.g. tree DP).
One pass

Greedy

One sentence: Make the best local choice at each step and hope those immediate jumps lead to the global best — often one loop, one variable, no table.

Real-life analogy

Climbing a hill in pitch-black fog. You can't see the summit — only your feet. Each step, pick the direction that goes steepest upward right now. Greedy never reconsiders past choices.

Interactive trace — Jump Game

From index i you can jump up to nums[i] steps. Track farthest — the rightmost index reachable so far.

Step 1 / ?

Minimal C skeleton

int canJump(int *nums, int numsSize) {
    int maxReachable = 0;

    for (int i = 0; i < numsSize; i++) {
        if (i > maxReachable) return 0;   /* stuck — can't reach i */

        int currentReach = i + nums[i];
        if (currentReach > maxReachable)
            maxReachable = currentReach;

        if (maxReachable >= numsSize - 1) return 1;
    }
    return 0;
}

Line-by-line trace

  1. Horizon: maxReachable = 0 — furthest index you can touch.
  2. Scan: if current index i is past that horizon, you're stuck → return 0.
  3. Local choice: update horizon with i + nums[i]; early exit once horizon passes the last index.

Interval problems (q117, q122) use the same spirit: sort by end time, keep the greedy choice, one pass.

Stack

Monotonic stack

One sentence: Keep a stack sorted in one direction (increasing or decreasing) to find the "next greater" or "next smaller" element in O(n) time.

Real-life analogy

People in a single-file line. A tall person joins at the back — they block the view of everyone shorter in front of them from a back-to-front perspective. The stack holds who still "matters" when scanning backward.

Diagram — Daily Temperatures

Minimal C skeleton (next greater — scan right to left)

void nextGreaterElement(int *nums, int size, int *result) {
    int stack[size];
    int top = -1;

    for (int i = size - 1; i >= 0; i--) {
        while (top >= 0 && stack[top] <= nums[i])
            top--;                          /* pop smaller — can't be next greater */

        result[i] = (top == -1) ? -1 : stack[top];
        stack[++top] = nums[i];
    }
}

Line-by-line trace

  1. Setup: array stack[] + top = -1 (empty stack).
  2. Backward scan: start at the last index — each step asks "what's the next greater to my right?"
  3. Pop smaller: while stack top ≤ current value, pop — those values are useless for anything left of here.
  4. Answer & push: stack top (if any) is the next greater; push current value for future indices.

Variant — Daily Temperatures (forward scan, index stack)

int stack[MAX], top = -1;

for (int i = 0; i < n; i++) {
    while (top >= 0 && T[i] > T[stack[top]]) {
        int prev = stack[top--];
        wait[prev] = i - prev;
    }
    stack[++top] = i;
}

Search

Backtracking

One sentence: Build a solution step by step; when a path breaks a rule, undo the last choice and try the next branch.

Real-life analogy

A ring of unlabelled keys on a locked door. Try one key, turn it, doesn't work — pull it back out and try the next. Backtracking is that choose → explore → unchoose loop in code.

How it differs from grid DFS

Grid DFS marks cells permanently (sink to '0'). Backtracking un-does each pick so another branch can reuse the same slot. Classic for subsets, permutations, N-queens.

Minimal C skeleton

void backtrack(int *nums, int size, int start,
               int *currentSet, int setSize) {
    printSubset(currentSet, setSize);   /* base: record current state */

    for (int i = start; i < size; i++) {
        currentSet[setSize] = nums[i];              /* CHOOSE */
        backtrack(nums, size, i + 1, currentSet, setSize + 1);  /* EXPLORE */
        /* UNCHOOSE — next loop overwrites slot; setSize shrinks on return */
    }
}

Line-by-line trace

  1. Base: every call records the current subset (including empty).
  2. Start index: start prevents duplicates — only pick forward in the array.
  3. Branch: append nums[i], recurse with i+1; when that returns, loop tries the next i.

Runtime cheat sheet

  • Binary search — O(log n) time, O(1) space on sorted arrays
  • Two pointers — O(n) time, O(1) space (one pass from both ends or slow/fast)
  • Sliding window — O(n) time, O(1) or O(k) space for frequency table
  • Hash / frequency — O(n) time, O(n) space for count arrays
  • Kadane (max subarray) — O(n) time, O(1) space
  • DFS / BFS on grid — O(rows × cols) time and visit array
  • DP tabulation — O(n) or O(n×m) time, O(n) space (often reducible)
  • Monotonic stack — O(n) time — each index pushed/popped once
  • Backtracking — often O(2n) or O(n!) — fine for small n in interviews

Firmware / drivers

Systems-level & low-level algorithmic patterns

Classic DSA assumes heap freedom and pointer-heavy structures. In kernel, driver, and firmware work, algorithms run under tighter rules: no malloc in hot paths, predictable latency, and memory that sits next to hardware (DMA rings, UART FIFOs, GPIO registers).

6.1 Zero-copy stream parsing

One sentence: Walk a fixed byte buffer with a cursor — expose slices as pointers and lengths instead of copying into new arrays.

Sliding window on strings is the interview version. On UART/SPI/Wi-Fi/BLE you parse frame headers, length fields, and CRC bytes in place over a DMA or ring buffer. No memcpy per field.

Core mechanics: dual pointers over static storage — sync word hunt, length check, advance cursor past frame boundary.

#include <stdint.h>
#include <stdbool.h>
#include <stddef.h>

typedef struct {
    uint16_t preamble;
    uint8_t payload_len;
    const uint8_t *payload;  /* points into DMA/ring buffer — no copy */
    uint8_t crc;
} zero_copy_frame_t;

bool parse_next_frame(const uint8_t *buffer, size_t buffer_len,
                      size_t *cursor, zero_copy_frame_t *out) {
    while (*cursor + 4 <= buffer_len) {
        uint16_t sync = (buffer[*cursor] << 8) | buffer[*cursor + 1];
        if (sync == 0xAA55) {
            uint8_t len = buffer[*cursor + 2];
            if (*cursor + 4 + len <= buffer_len) {
                out->preamble = sync;
                out->payload_len = len;
                out->payload = &buffer[*cursor + 3];
                out->crc = buffer[*cursor + 3 + len];
                *cursor += 4 + len;
                return true;
            }
            return false;  /* incomplete frame — wait for more bytes */
        }
        (*cursor)++;  /* slide 1 byte and hunt sync again */
    }
    return false;
}

6.2 Bitwise invariants & combinatorial state

One sentence: Replace fat lookup tables with bitmasks — one uint32_t can hold dozens of on/off flags.

Interviewers use this to test whether you think about cycles and RAM, not just big-O on paper.

  • Clear lowest set bit: x & (x - 1) — loop runs once per set bit, not per bit width.
  • GPIO / IRQ masks: enable, disable, or test groups of pins with OR/AND in one instruction.
  • Subset enumeration: iterate mask from 0 to (1<<n)-1 for small n instead of recursion.
#include <stdint.h>

/* Count set bits in O(number of set bits) — Brian Kernighan */
uint32_t count_active_flags(uint32_t status_register) {
    uint32_t count = 0;
    while (status_register > 0) {
        status_register &= (status_register - 1);
        count++;
    }
    return count;
}

6.3 Fixed-size circular ring buffers

One sentence: FIFO with a fixed array — head/tail wrap with modulo (or bitmask when size is power of 2).

Linked-list queues call malloc per node — bad for ISRs and long-running firmware (fragmentation + jitter). A ring buffer is O(1) push/pop with predictable memory.

Power-of-2 trick: if SIZE = 256, use (idx + 1) & (SIZE - 1) instead of % SIZE — cheaper in interrupt handlers.

#include <stdint.h>
#include <stdbool.h>

#define BUFFER_SIZE 256       /* must be power of 2 */
#define BUFFER_MASK (BUFFER_SIZE - 1)

typedef struct {
    uint8_t storage[BUFFER_SIZE];
    volatile uint32_t head;
    volatile uint32_t tail;
} lockless_ring_buffer_t;

bool ring_buffer_push(lockless_ring_buffer_t *buf, uint8_t data) {
    uint32_t next_head = (buf->head + 1) & BUFFER_MASK;
    if (next_head == buf->tail) return false;  /* full */
    buf->storage[buf->head] = data;
    buf->head = next_head;
    return true;
}

bool ring_buffer_pop(lockless_ring_buffer_t *buf, uint8_t *out_data) {
    if (buf->tail == buf->head) return false;  /* empty */
    *out_data = buf->storage[buf->tail];
    buf->tail = (buf->tail + 1) & BUFFER_MASK;
    return true;
}

6.4 Cache-conscious layouts (array heaps)

One sentence: Store trees and heaps in flat arrays so child nodes sit in contiguous cache lines — fewer pointer chases.

Pointer-based BST nodes scatter across the heap → cache misses dominate at scale. A binary heap (or level-order tree) in int heap[] uses index math only:

  • Left child of i → 2*i + 1
  • Right child of i → 2*i + 2
  • Parent of i → (i - 1) / 2

Same idea as heap sort and priority queues in embedded schedulers — no struct Node * required in the hot path.

Practice: q26 Heap sort q63 BST (pointer form) — compare with array indexing when you need speed.

Systems DSA interview checklist

State these out loud before you code — interviewers notice:

  1. No heap in hot path: "I'll avoid malloc/free here for deterministic timing and no fragmentation."
  2. Prefer iteration: "I'll use a loop instead of deep recursion so stack usage stays bounded in kernel/ISR context."
  3. Concurrency: "If this runs in an ISR, I'll use a lock-free ring or document where a memory fence is needed."
